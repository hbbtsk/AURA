# AURA 路线图

> 这是一份**技术演进计划**，而非产品发布时间表。
> 我们正在构建一个根植于叙事学的事件驱动叙事引擎。
> 路线图展示了架构如何从"单角色 Prompt 优化"深化为"多 Agent 持久世界模拟"。

---

## Phase 0：架构验证（当前）

**状态**：进行中  
**目标**：验证"确定性状态层 + 双输出架构"能解决纯 Prompt 工程无法解决的问题。

| 模块 | 交付物 | 状态 |
|------|--------|------|
| Event Patch 数据模型 | EventPatch schema、因果字段、可见性规则、事件否定定义 | 已验证 |
| 8层角色模型 | 体格/声纹/根脉/人际/核心/张力/轨迹/钩锚定义 | 已验证 |
| 双输出架构 | `narrative` + `structured` 单次调用输出 schema | 已定义 |
| Prompt 编译器骨架 | LangGraph 15节点工作流、TAVO API 兼容 | v0.8 可运行 |
| 3层记忆 | WORKING + RECENT + LONG_TERM RAG | 基础版 |
| 叙事学理论基础 | 布雷蒙序列、托多罗夫平衡、Trabasso 因果测试 | 已定义 |

**Phase 0 完成里程碑**：  
通过 AURA 的单角色 RP 会话，在受控测试中相比原生 ST（同卡、同模型、20+轮）展现出可量化的 consistency 提升。

## Phase 1：质量守卫层（v1.0.x）

**状态**：下一步  
**目标**：让 Mode A（Prompt 编译器）能用于单人 RP 生产环境。

**关键架构决策**：质量守卫是**输出后处理且非阻塞**的。双输出架构在单次 LLM 调用中同时产出 narrative 和 structured 数据。守卫在产出后检查；它们从不阻塞 SSE 流。问题被记录进状态并在下轮 Prompt 中修正 —— 不是通过重试当前 LLM 调用。

| 功能 | 技术方案 | 为什么重要 |
|------|----------|-----------|
| 越权检测 | SSE 流式传输期间对 `narrative.content` 做轻量正则（<1ms） | 防止模型写用户台词；热路径上唯一的零容忍检查 |
| 风格污染过滤器 | 与基线的声纹层指纹匹配（异步） | 捕获模型漂移回训练数据文风；将修正信号喂入状态供下轮使用 |
| 长度守卫 | 硬最小/最大 token 数 + 模板回退（异步） | 阻止模型啰嗦或回复只有一个字 |
| 意图标注器 v2 | 从双输出的 `structured.intent_tags` 中提取 | LLM 自我标注意图；替代启发式意图解析 |
| ST 卡导入器 | PNG 元数据解析器 → 8层 JSON 填充 | 桥接现有 ST 生态与 AURA 状态模型 |
| 非阻塞质量流水线 | 除越权检测外所有守卫通过事件总线异步运行 | 消除因质量检查导致的用户感知"卡顿" |

**里程碑**：用户可以导入 ST 卡，运行 30 轮会话，观察到的 OOC 事件比同后端模型的原生 ST 更少。会话全程不因质量检查而"冻结"。

## Phase 2：世界平台基础（v1.1.x）

**状态**：架构已验证，代码待写  
**目标**：从"一个角色 + 用户"走向"多个角色 + 世界规则"。

| 组件 | 职责 |
|------|------|
| Director | 场域快照渲染、提及解析、NPC 调度、冲突仲裁、序列推进 |
| NPC Agent | 每角色独立 System Prompt + 单次 LLM 调用 → 双输出，记忆过滤后的场域切片 |
| 世界状态管理器 | 原子性 EventPatch 应用、检查点存取、物理规则强制、`world_delta` 校验 |
| Cartridge 加载器 | YAML → Pydantic 解析器、一致性校验、多语言别名支持 |
| 序列层 | PresetSequence（Galgame）+ DynamicSequence（开放世界）管理 |

**关键技术挑战**：  
并发 NPC LLM 调用（2-3 个角色响应同一事件）而不产生指数级 token 成本。解决方案：共享上下文检索 + 每 Agent Prompt 切片 + 用于 Director 仲裁的双输出结构化数据。

**关键架构决策 —— 可见性**：  
每个 NPC 对每一个事件维护独立的可见性映射。同一个 EventPatch 为每个 NPC 携带不同的 `character_delta` 和 `visibility`。"Pyrrha 知道什么"和"Weiss 知道什么"由 Director 独立计算。

**里程碑**：3 角色场景（玩家 + 2 NPC）运行 10 轮无串味，NPC 正确引用彼此之前的陈述。预设序列能基于状态条件推进。

## Phase 3：因果引擎（v1.2.x）

**状态**：已定义  
**目标**：让长弧叙事在跨会话中保持一致。

| 功能 | 方案 |
|------|------|
| 因果图存储 | Kuzu 图数据库用于 `triggered_by` / `causes` 链接 |
| 因果测试（Trabasso）| "如果没有 A，就没有 B" —— 自动化因果链接验证 |
| 序列结构 | 布雷蒙基本序列 + 复合连接（连接式/镶嵌式/两面式）|
| CausalRAG | 检索的不是"相似事件"而是"因果相关事件" —— 因果链优先，嵌入相似度其次 |
| 根因追踪 | 每个事件都知道其终极起源；防止剧情倒退 |
| 会话检查点 | 保存世界状态 + 事件图 + 序列进度；精确恢复到离开时的状态 |

**里程碑**：跨越 5 次会话（总计 50+ 轮）的悬疑剧情线保持线索一致性；红鲱鱼不会意外成真，真线索不会消失。闪回序列（镶嵌式）正确融入主时间线。

## Phase 4：事件涌现（v1.3.x）

**状态**：计划中  
**目标**：世界在无玩家直接输入的情况下生成事件。

| 引擎 | 功能 |
|------|------|
| EventEngine | NPC 基于目标和状态安排幕外行动；DynamicSequence 生成 |
| PacingEngine | 通过序列结构监控叙事张力；注入缓和或升级 |
| PerturbationEngine | 随机世界事件（天气、新闻、事故）迫使角色反应；应用前由世界状态管理器校验 |

**里程碑**：玩家 24 小时后登录；2-3 个"幕外事件"已经发生，改变了 NPC 情绪状态和可用对话话题。事件与玩家先前行动因果关联（通过 Trabasso 测试）。

---

## Phase 5：多 Agent 并发（v1.4.x）

**状态**：计划中  
**目标**：扩展到 5+ 同时在线 NPC，具有有意义的群体动态。

| 问题 | 解决方案 |
|------|----------|
| 指数级 LLM 成本 | 批量上下文检索；共享场域快照；选择性 NPC 激活；双输出降低每 NPC 开销 |
| 冲突检测 | 当两个 NPC 提出矛盾的 `world_delta`，Director 按优先级 + 时间戳 + 叙事功能仲裁 |
| 离线模拟 | NPC 通过 EventEngine 在后台线程中继续"生活"，玩家离开时生成事件 |
| 两面式序列 | 同一事件对不同角色有不同意义；Director 计算独立影响 |

**里程碑**：5 NPC + 玩家的酒馆场景；NPC 有旁侧对话、偷听、打断或无视玩家，基于注意力过滤器。可见性规则确保每个 NPC 只知道它应该知道的内容。

## 叙事学集成时间线

叙事学不是后期添加的"功能"。它是随每个阶段演进的**基础**：

| 阶段 | 叙事学集成 |
|------|-----------|
| v0.9.x | 事件否定定义（主体 + 行为 + 影响）；非事件过滤 |
| v1.0.x | 意图标注作为叙事功能标记；声纹层作为巴尔特的"声音" |
| v1.1.x | 序列层：基本序列（布雷蒙）；预设 vs 动态 |
| v1.2.x | 完整因果图：Trabasso 测试、复合连接、CausalRAG |
| v1.3.x | 涌现叙事：托多罗夫平衡模型用于世界状态节奏控制 |
| v1.4.x | 多 Agent：两面式序列、每角色可见性作为叙事聚焦 |

## 驱动路线图的设计原则

1. **状态先于文本**：物理/心理状态变化由规则计算；LLM 只负责认知/表达层。
2. **因果先于相似**：RAG 优先按因果链检索，嵌入相似度其次。
3. **不 LLM 重试**：输出守卫过滤或截断，从不请求 LLM 重新生成。问题通过状态在下轮 Prompt 中修正。
4. **玩家自带密钥**：我们不托管模型；我们托管架构。
5. **文本为根**：所有叙事逻辑可检查、可 diff、可版本控制。
6. **叙事学驱动工程**：每个技术决策都可追溯到一个叙事学概念。
