# AURA 路线图

> **这是技术路线演进计划，不是产品发布时间表。**  
> 我们正在构建事件驱动的叙事引擎。本路线图展示架构如何从"单角色 Prompt 优化"深化到"多智能体持久世界模拟"。

---

## 阶段 0：架构验证（当前）

**状态**：🚧 进行中  
**目标**：证明"事件总线 + 状态机"能解决纯 Prompt 工程无法解决的问题。

| 模块 | 交付物 | 状态 |
|------|--------|------|
| 事件总线数据模型 | EventPatch 模式、因果字段、可见性规则 | ✅ 已验证 |
| 八层人物模型 | 存在/腔调/根底/脉络/内里/张力/轨迹/钩子定义 | ✅ 已验证 |
| Prompt 编译器骨架 | LangGraph 15 节点工作流，TAVO API 兼容 | 🚧 可运行 |
| 三层记忆 | WORKING + RECENT + LONG_TERM RAG | 🚧 基础版 |

**阶段 0 完成里程碑**：  
在受控测试中，通过 AURA 的单角色 RP 会话与原生 ST（同一张卡、同一个模型、20+ 轮）相比，展现出可量化的一致性提升。

## 阶段 1：质量校验层（v1.0.x）

**状态**：📋 下一步  
**目标**：让 Mode A（Prompt 编译器）在单人 RP 场景下达到可用状态。

| 功能 | 技术方案 | 为什么重要 |
|------|---------|-----------|
| 越权检测 | 正则 + JSON Schema 后过滤，不调用 LLM 重试 | 防止模型代写用户台词 |
| 文风污染过滤 | 与基线进行腔调层指纹匹配 | 捕捉模型滑回训练数据腔调 |
| 长度守卫 | 硬最小/最大 token + 模板兜底 | 阻止模型啰嗦或只回一个字 |
| 意图标签器 v2 | 轻量分类器（本地 7B 或 Embedding） | 用结构化标签替代启发式解析 |
| ST 角色卡导入器 | PNG 元数据解析 → 八层 JSON 填充 | 打通现有 ST 生态与 AURA 状态模型 |

**里程碑**：用户导入一张 ST 卡，运行 30 轮会话，观察到比原生 ST（同后端模型）更少的 OOC 事故。

## 阶段 2：世界平台基础（v1.1.x）

**状态**：📋 架构已验证，代码待实现  
**目标**：从"一个角色 + 用户"进化到"多角色 + 世界规则"。

| 组件 | 职责 |
|------|------|
| Director | 场域快照渲染、提及解析、NPC 调度、冲突仲裁 |
| NPC Agent | 每个角色的独立 System Prompt + 隔离 LLM 调用，记忆过滤后的场域切片 |
| 世界状态管理器 | 原子化 EventPatch 应用、检查点存取、物理规则强制执行 |
| Cartridge 加载器 | YAML → Pydantic 解析器、一致性校验、多语言别名支持 |

**关键技术挑战**：  
并发 NPC LLM 调用（2-3 个角色响应同一事件）而不产生指数级 token 成本。解决方案：共享上下文检索 + 每 Agent Prompt 切片。

**里程碑**：一个 3 角色场景（玩家 + 2 个 NPC）运行 10 轮无串戏，NPC 能正确引用彼此之前的陈述。

---

## 阶段 3：因果引擎（v1.2.x）

**状态**：📋 计划中  
**目标**：让长线叙事跨会话保持一致。

| 功能 | 方案 |
|------|------|
| 因果图存储 | Kuzu 图数据库存储 `triggered_by` / `causes` 链接 |
| CausalRAG | 检索不只是"相似事件"，而是"因果相关事件" |
| 根因追踪 | 每个事件知道其终极起源；防止剧情回退 |
| 会话检查点 | 保存世界状态 + 事件图；精确恢复到离开时的状态 |

**里程碑**：一个跨越 5 次会话（总计 50+ 轮）的悬疑剧情保持线索一致性；红鲱鱼不会意外成真，真线索不会消失。

---

## 阶段 4：事件涌现（v1.3.x）

**状态**：📋 计划中  
**目标**：世界在无玩家直接输入时自行产生事件。

| 引擎 | 功能 |
|------|------|
| EventEngine | NPC 基于目标和状态安排幕后的行动 |
| PacingEngine | 监控叙事张力；注入缓和或升级 |
| PerturbationEngine | 随机世界事件（天气、新闻、意外）迫使角色反应 |

**里程碑**：玩家 24 小时后登录；2-3 个"幕后事件"已发生，改变 NPC 情绪状态和可用对话话题。

---

## 阶段 5：多智能体并发（v1.4.x）

**状态**：📋 计划中  
**目标**：扩展到 5+ 同时 NPC，具备有意义的群体动力学。

| 问题 | 解决方案 |
|------|---------|
| 指数级 LLM 成本 | 批量上下文检索；共享场域快照；选择性 NPC 激活 |
| 冲突检测 | 当两个 NPC 提出矛盾的 world_delta 时，Director 按优先级 + 时间戳仲裁 |
| 离线模拟 | NPC 在后台线程中继续"生活"，在玩家离线时生成事件 |

**里程碑**：一个酒馆场景，5 个 NPC + 玩家；NPC 有旁支对话、偷听、打断或基于注意力过滤无视玩家。

## 驱动路线图的设计原则

1. **状态先于文本**：物理/心理状态变更由规则计算；LLM 只处理认知层。
2. **因果先于相似**：RAG 先按因果链检索，其次才是 Embedding 相似度。
3. **不打回 LLM**：输出守卫过滤或截断，绝不要求 LLM 重新生成。
4. **用户自带 Key**：我们不托管模型；我们托管架构。
5. **文本即根**：所有叙事逻辑可检查、可对比、可版本控制。
